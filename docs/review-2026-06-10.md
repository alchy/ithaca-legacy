## 1. Nálezy nejvyšší priority (napříč oblastmi)

### 1.1 [KRITICKÁ] Lazy build harmonické matice 128×128 na audio vlákně
`engine/resonance/harmonic_proximity.cpp:50–66` — `couplingMatrix()` je function-local
static; jediný volající je `ResonanceEngine::onPlayedNoteOn` (audio vlákno). Build
= ~4,2 M iterací s `log2f/expf/powf` → stovky ms (RPi5 až ~1 s) v audio callbacku
při **první zahrané notě každé session** = deterministický dropout.
Doc E-resonance.md:89 tvrdí „first call: off-RT lazy init“ — žádný off-RT init neexistuje.
**Fix (triviální):** warm-up v `Engine::init` — `(void)harmonicProximity(0, 1);`.

### 1.2 [KRITICKÁ] Non-RT logger (mutex + fflush + alokace + subscriber) v audio hot-path
Nejrozšířenější systémový problém, potvrzen 5 oblastmi (A, C, D, E, I):
- `engine/engine.cpp:300, 321, 337, 353, 376` — log při každém NoteOn/NoteOff/CC64/bloku s rezonancí
- `engine/voice/voice_pool.cpp:102, 111` — voice_on/voice_steal
- `engine/voice/voice.cpp:81, 184–197, 276–284` — prepareDamp/DEACTIVATE/UNDERRUN/EOF

`Logger::vlog` (log.cpp:121–148) bere `log_mutex_` (pod ním `fputs`+`fflush`),
`subscriber_mtx_`, alokuje 2× `std::string`; GUI subscriber bere třetí mutex.
Po zavedení RT priority (`enableRealtimeAudio`, SCHED_FIFO) jde o **neomezenou
priority inversion** — std::mutex nemá priority inheritance. Porušuje vlastní
dokumentovaný kontrakt (A-core.md: „audio vlákno nikdy nezamyká mutex“).
**Fix:** převést na existující `LOG_RT_*` (lock-free ring funguje, ResonanceVoice ho už používá),
nebo compile-time gate debug logů.

### 1.3 [KRITICKÁ] Recyklace stream ringu s in-flight worker requestem (ABA / 2 producenti)
`engine/stream/stream_engine.cpp:108–142, 158–229` + `voice.cpp:75–77, 350–368`
(potvrzeno nezávisle oblastmi C, D, I). `releaseRing` z audio vlákna nečeká na
dokončení worker requestu a request nelze odvolat; `acquireRing` skenuje od slotu 0
→ při steal/retriggeru je tentýž ring okamžitě znovu přidělen:
1. `resetForReuse()` (w_=r_=0) souběžně s worker `push()` → data race na kurzorech (UB);
2. nový hlas pošle vlastní request → **dva workeři píší do téhož ringu** → propletená data dvou souborů;
3. starý request s `eof_when_done` nastaví `eof_` ringu nového hlasu → předčasný konec uprostřed samplu.

Druhá cesta: `stream_pending_` se shodí při `avail >= half_cap` dřív, než request
doběhl (pomalý disk) → druhý request na tentýž ring.
Komentář v `releaseRing` („nic strašného, jen plýtvání“) re-acquire nepokrývá;
C-buffers.md sekce „Nálezy revize: bez nálezů“ deklaruje SPSC bezpečnost, která neplatí.
**Fix:** `std::atomic<uint32_t> generation_` per ring, inkrement v releaseRing;
`StreamRequest` nese snapshot, worker ověří před každým push i před `eof_.store`.

### 1.4 [VYSOKÁ] Synchronizace spánkem místo handshake — trojice protokolů (UAF riziko)
(potvrzeno oblastmi A, E, F, I)
- **reloadBank** (`engine.cpp:123–152`): `bank_loading_.store(true)` → `sleep(10ms)` →
  `reset()` + přepis `bank_`. Audio čte flag jen na začátku bloku; `setBlockSize`
  povoluje 8192 frames = perioda ~170 ms → in-flight blok může přežít sleep a číst
  uvolněnou banku (use-after-free). GUI reload navíc blokuje render smyčku na sekundy.
- **rebuildResonanceCache** (`engine.cpp:537–568`): `clearCacheReady` + fade request →
  `sleep(120ms)` → `buildResonanceCache` **realokuje** `preload_resonance` pod čtoucím
  audio vláknem. Fade request se konzumuje až v příštím `processBlock` — při bloku
  ≥ 4096 se 120 ms nemusí stihnout.
- **Coalesce větev recache** (`engine.cpp:550–567`): druhá iterace běží **bez nového
  fade requestu** — mezitím `setCacheReady` znovu povolil spawn cache-mód hlasů (žijí
  až 12 s) → iterace 2 dealokuje buffer pod čtoucím hlasem. Nejostřejší varianta díry.
- **Lost-update na `recache_has_pending_`**: GUI nastaví pending poté, co worker už
  prošel exchange → pending=true, žádné vlákno neběží, cache ready shozená → rezonance
  trvale ve stream módu do dalšího pohybu slideru.

**Fix:** epoch/ack handshake — audio publikuje `atomic<uint64_t> block_epoch_` na konci
bloku, reload/recache čekají na inkrement; recache stav {idle,running,pending} pod
jedním mutexem (non-RT cesta); fade+clear před každou iterací rebuildu.

### 1.5 [VYSOKÁ] Damping crossfade osiří při jiném slotu → retrigger/steal NENÍ click-free
`engine/voice/voice_pool.cpp:179` + `voice.cpp:84` — `prepareDamp` deaktivuje hlas
(`active_=false`, `damping_=true`), ale `VoicePool::processBlock` dampované hlasy
skipuje (`if (!v.active()) continue;`). Damp ocas se přehraje jen pokud `findSlot`
náhodou vrátí tentýž slot. Jinak: (a) starý průběh noty tvrdě uříznut = lupnutí
(běžný případ při volných nižších slotech); (b) osiřelý `damp_buf_` zdědí pozdější
nesouvisející noteOn → 21ms „duch“ staré noty. `Voice::process` je na dohrávání
po deaktivaci stavěný — větev je mrtvý kód. D-polyphony.md:39 popisuje opak reality.
**Fix (jednořádkový):** v processBlock volat `v.process()` i pro `isDamping()` hlasy.

### 1.6 [VYSOKÁ] MidiQueue: SPSC fronta má fakticky více producentů
`engine/midi/midi_queue.h:27–34` — producenti: RtMidi callback vlákno, GUI vlákno
(`reloadBank → allNotesOff`, panel_bank.cpp:60), CLI main. Dva souběžné `push()`
→ ztracený event + formální data race. Navíc `allNotesOff`/`sustainPedal` ignorují
návratovou hodnotu push (tichá ztráta pedal-up/panic při plné frontě), zatímco
noteOn/noteOff selhání logují.
**Fix:** CAS na `w_` (producenti jsou non-RT), nebo sticky atomic pro AllNotesOff/Sustain
mimo frontu; minimálně logovat drop.

### 1.7 [VYSOKÁ] Convolver: 2-slot IR swap bez potvrzení od audio vlákna
`engine/dsp/convolver.cpp:33–41` — `setIR` přepisuje neaktivní slot; při dvou
publikacích během jednoho in-flight `process()` (init burst dělá 3–4 swapy během
~100 µs; dnes kryje jen to, že běží před `audio->start`) přepíše slot, který audio
právě čte → latentní use-after-free. Žádný epoch/ack mechanismus.
**Fix:** audio po process() potvrdí viděný index (`seen_.store`), setIR přepisuje
jen potvrzeně opuštěný slot; nebo RCU s odloženou dealokací.

### 1.8 [VYSOKÁ] Rezonanční budget gate: srovnává cíl s okamžitým gainem čerstvých spawnů
`engine/resonance/resonance_engine.cpp:178–196` — čerstvě spawnutý hlas má `gain_=0`
(rampuje 30 ms) → při plném budgetu každé další eligibilní N vyhodnotí `qlevel==0`
→ fade právě spawnutého + spawn nového = spawn-churn (disková čtení), přesně to,
co měl gate eliminovat; přežije poslední N, ne nejhlasitější. E-resonance.md:198
(„srovnání na stejné škále“) je nesprávný závěr.
**Fix:** porovnávat `max(currentLevel(), targetGain())`.

### 1.9 [VYSOKÁ] GUI vlákno: per-frame enumerace MIDI a synchronní reload banky
- `app/gui/panel_topbar.cpp:29` — `MidiInput::listPorts()` každý frame konstruuje
  `RtMidiIn` (OS MIDI klient, IPC) @ 60 Hz; tlačítko RESCAN je no-op. Fix: cache + rescan
  na tlačítko/otevření comba.
- `app/gui/panel_bank.cpp:60,99` — `reloadBank` synchronně v render smyčce: sleep 60 ms
  + **celý diskový load** (sekundy až desítky sekund; recache čte ~200 MB) → okno
  „neodpovídá“. H-gui.md uvádí jen „~60 ms“ — výrazně podceněno. Fix: worker thread +
  progress (engine má `bank_loading_` flag, loadBank potřebuje progress callback).

## 4. Testy — hlavní mezery v pokrytí

Jednotky jsou pokryté výborně, integrační a chybové dráhy téměř vůbec:
1. **Voice stealing zcela nepokryt** — fáze 2–4 findSlot bez jediného testu; noteOn
   s pedálem se v testech nevolá; retrigger test používá jedinou notu (nemůže odhalit 1.5).
2. **Pedál × pool integrace nulová** — noteOffWithPedal/releasePendingNotes/
   markPendingRelease: 0 zásahů v tests/; sustain dráha přes Engine::processBlock netestována.
3. **Rezonanční budget/steal/half-pedal** — žádný test s cap < počet harmonik
   (odhalil by 1.8); half-pedal jen 127/0.
4. **Streaming chybové dráhy** — recyklace ringu s in-flight requestem (odhalil by 1.3),
   drop-on-full, skutečný underrun (tvar výstupu), souběžný stress SP-MC fronty.
5. **Loader malformed vstupy** — lhající data_size, frames==0, bits=8, EXTENSIBLE;
   24/32-bit dekódovací dráhy bez testu (ruční sign-extend!).
6. **Persistence chybové dráhy** — korupce čísel, useknutý soubor, v3 migrace,
   bbe_* → enhancer_* migrace.
7. **Threading** — žádný TSan/stress test v CI; build-tsan existuje, nabízí se
   MIDI-flood + reload + slider scénář.
8. Convolver: MIX early-out kontinuita (ověřena ad-hoc, bit-exact — zakotvit jako
   regresní test), setIR > kMaxIr, stale-history po re-enable.

## 5. Doporučené pořadí oprav

1. **RT hygiena** (1.1 warm-up matice + 1.2 logy → LOG_RT_*) — největší dopad,
   triviální až malé změny; odblokuje smysl RT priority.
2. **Damping fix** (1.5) — jednořádková oprava, slyšitelné zlepšení retrigger/steal.
3. **Ring generation counter** (1.3) — lokální fix, odstraní audio korupci + UB.
4. **Handshake protokoly** (1.4: reloadBank epoch ack; recache stav pod mutexem
   + fade před každou iterací) — odstraní UAF třídu.
5. **Budget gate** (1.8) + MidiQueue MPSC (1.6) + convolver swap ack (1.7).
6. **GUI**: async reload + progress, MIDI port cache, persistence error-paths.
7. **Výkon (RPi5)**: paralelní loader, popFrame bulk čtení, SoA voice pool,
   convolver partitioned FFT (volitelné — dnes 4 % CPU na M1).
8. **Dokumentační úklid** dle §3 (jedním průchodem, vč. uzavření vyvrácených nálezů
   F-loader #2 a H-gui #4).
