# Polyfonie

Ithaca udrzuje pool az 128 (max 256) nezavislych hlasu (`Voice`), z nichz kazdy prehrava jeden `SampleAsset` pres prvni mikrofon (`mics[0]`). `VoicePool` rozhoduje o alokaci, retrigger-dampovani a kradenI hlasu; `patch_manager` vybira konkretni `SampleAsset` a velocity-gain pro danou (nota, velocity) kombinaci. Hlas sam riadi tri navrstene envelopy (onset anti-click, release, underrun-fade), damping crossfade pri retriggeru/kradenI a linearni interpolaci pri prehravani z preload-head i streamovaneho ringu. Streaming (vlastnictvi ringu, lo/hi interpolacni okno, refill heuristika, bulk cteni ringu) zije ve sdilenem `StreamedSampleReader` (`engine/stream/streamed_reader.{h,cpp}`, viz [C-buffers](C-buffers.md)) — Voice drzi jen POLICY pri prazdnem ringu (cisty konec vs underrun). Damp buffery vsech hlasu lezi v souvislem `VoicePool::damp_pool_` (Voice drzi jen pointer) — `sizeof(Voice)` je diky tomu ~stovky B misto ~16,5 kB.

(5) Pri deaktivaci `reader_.release(stream_)`. Navratova hodnota: `active_ || damping_ || produced` — hlas zustava zpracovavany dokud dozniva damping crossfade i po deaktivaci (`VoicePool::processBlock` dampujici hlasy nepreskakuje). Vsechny deaktivacni/underrun logy jdou pres `LOG_RT_*` (lock-free RT ring), nikdy pres non-RT logger. |

## Krizove odkazy

| Oblast | Vazba |
|--------|-------|
| **Events — noteOn/Off zdroj** | `Engine::processBlock` drainuje lock-free MIDI frontu a vola `selectVoice` + `VoicePool::noteOn` / `noteOffWithPedal` / `releasePendingNotes` / `allNotesOff`. (`VoicePool::noteOff` je test-only API — Engine ho nevola.) Hlasy tedy vznikaji vzdy na audio threadu, i kdyz MIDI event pridal jiny thread. |
| **Buffers — streaming ring** | Veskery streaming jde pres sdileny `StreamedSampleReader` (`reader_` member, viz [C-buffers](C-buffers.md)): `Voice::start` → `reader_.begin` (acquire ring + prvni request); `Voice::process` → `reader_.beginBlock`/`endBlock` (bulk cteni: 1 acquire + 1 release za blok misto per-frame `popFrame`) + `reader_.refill` (requestRead vraci bool, pri plne fronte se offset neposouva); `Voice::prepareDamp` a `hardStop` vraceji ring (`reader_.release` → `releaseRing` — dela gen bump, viz oblast C). Ring je sdileny zdroj — main pool ma `EngineConfig::num_rings` slotu (default 256, vzdy ≥ poctu hlasu); pri selhani acquire (`begin` vrati false) hlas degraduje na preload-only rezim. |
| **Loader — Bank / sample_types** | `selectVoice` prijima `const Bank&` (struktura `NoteSlots`, `VelocitySlot`, `SampleAsset` definovane v `engine/sample/sample_types.h`). `Voice::start` uklada `const SampleAsset*` a `const MicLayer*` (mics[0]); nasledne cte `mic_->preload_head`, `mic_->head_frames`, `mic_->file.frames`, `mic_->file.sample_rate`, `mic_->file.path`, `mic_->mode`. ## Nalezy revize

### Osirely damping crossfade (retrigger/steal NENI click-free) — ✅ VYRESENO (fix/revize-2026-06-10)

Puvodni nalez (revize 2026-06-10, §1.5, VYSOKA): `prepareDamp` deaktivoval hlas (`active_ = false`, `damping_ = true`), ale `VoicePool::processBlock` dampovane hlasy skipoval (`if (!v.active()) continue;`). Damp ocas se prehraval jen kdyz `findSlot` nahodou vratil tentyz slot. Jinak: (a) stary prubeh noty byl tvrde uriznut = lupnuti; (b) osirely `damp_buf_` zdedil pozdejsi nesouvisejici noteOn na tom slotu → az 21ms "duch" stare noty.

Reseni: novy getter `Voice::isDamping()` a podminka v `VoicePool::processBlock` — preskakuji se jen hlasy, ktere nejsou `active()` ANI `isDamping()`. Damp ocas po `prepareDamp` tak dozni VZDY a v realnem case, i kdyz novy hlas dostal jiny slot. Tim mizi i "ghost" scenar: ocas se nedrzi zamrzly do pristiho noteOn, ale prehraje se hned (≤ 21 ms po retriggeru/stealu); pokud noteOn dorazi na tentyz slot jeste behem doznivani, ocas pokracuje soubezne s novym hlasem — coz je presne zamysleny crossfade. `start` damp stav resetovat nemusi.

### Underrun rozlisovani — heuristika `file_request_off_` — ✅ VYRESENO (fix/revize-2026-06-10)

Puvodni nalez: cisty konec vs. skutecny underrun se rozlisuje porovnanim `file_request_off_ >= total_frames`; `file_request_off_` se ale navysoval i pri dropu requestu (plna fronta) → skutecny underrun se mohl klasifikovat jako "cisty konec" (zadny `noteUnderrun()`, framy se uz nikdy nedozadaly).

Reseni: `StreamEngine::requestRead` vraci `bool`; pri `false` (drop-on-full) se offset NEPOSOUVA a refill heuristika request prirozene zopakuje pristi blok (logika dnes zije ve sdilenem `StreamedSampleReader::begin`/`refill`). `file_request_off_` tak roste jen o skutecne zarazene requesty a heuristika je spolehliva (`cleanEnd()`). Chovani pri rozliseni: cisty EOF → deactivate+zero s 5ms fade (Info END-OF-SAMPLE, reference icr); skutecny underrun → drzi posledni znamy vzorek a 5ms fade ho tvaruje (Warning + `noteUnderrun()`).

### Vykonovy nalez — 16 kB damp buffer inline, O(pool) `hasActiveMainVoice`, 3 atomiky na frame — ✅ VYRESENO (feat/async-loader-a-stream-refactor)

Puvodni nalez (revize 2026-06-10, §2 oblast D, STREDNI): (a) `sizeof(Voice)` = 16 528 B (16 kB `damp_buf_` inline) → vsechny skeny poolu (`processBlock`, `findSlot`, citace) krokovaly cache po ~16,5 kB; (b) `hasActiveMainVoice` byl O(pool) scan volany az 127× na jeden note-on z rezonancni eligibility; (c) hot-path cteni ringu delalo 3 atomicke operace na KAZDY frame v `popFrame`.

Reseni (vetev `feat/async-loader-a-stream-refactor`):
- (a) damp buffery presunuty do souvisleho `VoicePool::damp_pool_` — Voice drzi jen pointer (`setDampBuffer`, pointery pridelene po resize, zadna realokace) → `sizeof(Voice)` ~stovky B; test v `test_voice_pool.cpp` hlida `sizeof(Voice) < 512`.
- (b) per-nota citac `note_active_count_[128]` → `hasActiveMainVoice` O(1); ekvivalencni test proti ground-truth scanu v `test_voice_pool.cpp`.
- (c) bulk cteni ringu: `RingHandle::snapshotW/cursorR/commitR` + `StreamedSampleReader::beginBlock/endBlock` — 1 acquire + 1 release za blok, bit-exact shodne s per-frame `popFrame` (viz [C-buffers](C-buffers.md)); bit-exact chovani cele render cesty hlida `tests/test_render_regression.cpp`.

### `processBlock` ignoruje `engine_sr` parametr

`VoicePool::processBlock` prijima `engine_sr` ale okamzite provede `(void)engine_sr` a nepredava ho do `Voice::process`. `Voice::process` `engine_sr` fakticky nepotrebuje (vse per-frame je vypocteno pri `start`/`release`), takze to neni bug — ale signaturu lze zjednodusit a zbavit se mrtvého parametru.

### Envelope sekvencovani — release-v-onsetu spojity (aktualizovano fix/revize-2026-06-10)

Onset, release a underrun-fade jsou nezavisle rampy nasazovane multiplikativne. `release` startuje VZDY z `rel_gain_ = 1.0` a onset rampa dobiha dal — soucin onset×release je spojity (drive `rel_gain_ = onset_gain_` pri release-v-onsetu → v `process` se nasobily obe rampy → skok g0 → g0² = klik pri staccatu; opraveno). Pridany guard `release_ms ≥ 0.1` (0 by dalo −inf step). `prepareDamp` snima do damp bufferu obalku vcetne `onset_gain_` (steal behem nabehu bez skoku). Pedal pending-release je pri retriggeru spravne cisteny (`pending_release_ = false` ve `start`).
